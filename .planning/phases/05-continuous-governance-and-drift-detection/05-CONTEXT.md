# Phase 5: Continuous Governance and Drift Detection - Context

**Gathered:** 2026-04-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the loop on **governance**: merge requests must prove policy and integrity gates **before** landing on `main` (POL-03); **local and CI** run the same deterministic security scripts with identical pass/fail semantics (GOV-01); **requirement → phase** mapping stays machine-checkable without manual spreadsheet reconciliation (GOV-02); and **drift** between `security/security-policy.json`, documented entrypoints, and enforcement code surfaces as a failing check (GOV-03). This phase does not replace human security review or add hosted dashboards — it adds **scripts + CI wiring** aligned with existing Bash/Node tooling.

</domain>

<decisions>
## Implementation Decisions

### Merge gates and parity (POL-03, GOV-01)
- **D-01:** Add a **`pull_request`** GitHub Actions workflow (separate from Pages deploy) that runs **without** Claude/network-dependent eval generation by default: `scripts/verify-policy`, `node scripts/verify-artifact-integrity.mjs examples`, `node scripts/scan-publish-scope.mjs examples`, shell syntax checks (`bash -n evals/run.sh`), and Node `--check` on gated `evals/**/*.mjs` and `security/**/*.mjs` as listed in workflow.
- **D-02:** Document that **full** `bash evals/run.sh --subset --skip-llm` remains **maintainer-local** or optional scheduled job — PR gate stays fast and deterministic; POL-03 satisfied by **policy + integrity + scan** parity with what maintainers can run via `npm`-free commands in README.
- **D-03:** Same step order locally as in CI (document one block in README under “CI parity”).

### Traceability (GOV-02)
- **D-04:** Ship **`scripts/verify-requirements-trace.mjs`** that reads `.planning/REQUIREMENTS.md` traceability table and asserts each **v1** requirement row has **Phase N** and **Done | Pending** consistent with optional markers file `.planning/traceability.state.json` OR validates **regex-parseable** table rows only — minimum bar: **every INTG/DATA/RUN/POL/GOV id appears exactly once** and phase column matches `.planning/ROADMAP.md` phase list by title keyword match (simple heuristic) OR manual manifest `.planning/requirements-mapping.json` maintained by script scaffold — **decision:** start with **strict table validation** (duplicate IDs, orphan phases) plus optional JSON manifest generated by hand once; avoid heavy NLP.

Refined for implementability:
- **D-04b:** Implement verifier that (1) parses `## Traceability` markdown table for requirement IDs, (2) ensures each ID from `REQUIREMENTS.md` v1 bullet list appears in the table, (3) fails on duplicate rows. Phase alignment checked against a small **`scripts/traceability-allow.json`** mapping `REQ-ID → phase number` committed to repo — single source extended when roadmap changes.

### Drift detection (GOV-03)
- **D-05:** Add **`scripts/verify-policy-drift.mjs`** that loads `security/security-policy.json` and asserts **`entrypoints.gatedRelPaths`** files exist and that **`security/policy-runtime.mjs`** is the shared importer for those entrypoints (lightweight static grep or explicit export list in a manifest JSON generated by prior phase — **decision:** verify **file existence** + **grep** `assertEntrypointForCurrentModule` / `assertPublishArtifactPathAllowed` usage in listed scripts where applicable; document gaps instead of false positives).
- **D-06:** CI runs drift script on PRs after trace script.

### Claude's Discretion
- Exact job matrix (single ubuntu job vs matrix node versions) — default single Node 20 to match Pages workflow.
- Filename for traceability allowlist (`traceability-allow.json` vs embedded in script).

</decisions>
